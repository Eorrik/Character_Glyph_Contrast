# 项目合理性检查与执行计划

## 1. 结论摘要

基于 `project.md` 的当前定义，本项目作为一个**字形图像对比原型（prototype）**是总体合理的，原因如下：

1. **问题边界清晰**：输入限定为两张静态字形图像，不涉及联机书写轨迹，因此将问题建模为空间几何匹配是合理的。
2. **技术路线可落地**：`二值化 → 对齐归一化 → skeletonize → EDT → 差异可视化` 的流程已有成熟算法和 Python 生态支持。
3. **输出具备解释性**：骨架叠加图负责表达结构偏移，EDT 热力图负责表达粗细与局部形态差异，两者互补。
4. **适合作为首版原型**：Plotly/Dash 可以较快构建一个可交互的网页界面，便于验证方法有效性。

但如果要让原型真正稳定可用，`project.md` 还需要补足若干关键定义，尤其是**预处理规则、归一化策略、评价指标、数据约束与交互范围**。

---

## 2. 项目合理性检查

### 2.1 合理之处

#### 2.1.1 问题建模正确

项目明确只处理离线字形图像，不处理笔顺、速度、压力等时间序列信息。因此：

- 使用骨架表示结构中心线；
- 使用 EDT 表示局部笔画厚度与墨迹分布；
- 使用几何差异而不是时序建模；

这一建模方式与输入条件匹配，逻辑成立。

#### 2.1.2 模块拆分自然

当前拆分为：

- Structure Module（骨架）
- Ink Module（EDT）
- Visualization Module（网页展示）

这种拆分有利于并行开发，也方便后续替换算法，例如后续可将简单对齐升级为更稳健的仿射配准，或将 Hausdorff 替换为 Chamfer/对称 Chamfer。

#### 2.1.3 输出设计具备可解释性

- 骨架叠加图适合看“字形结构是否偏移”；
- EDT 热力图适合看“笔画厚薄和边界差异”；
- 网页交互适合看局部细节。

对于原型验证阶段，这是合理且有说服力的输出组合。

---

### 2.2 当前方案中的缺口与风险

#### 2.2.1 “空间归一化”定义不足

`project.md` 中提到了“尺度 + 对齐”，但没有说明具体如何做。这是全项目成败的关键之一。

建议明确以下规则：

- 是否先裁剪前景外接框（bounding box）；
- 是否统一缩放到固定画布尺寸（如 `256x256` 或 `512x512`）；
- 对齐依据是质心、边界框中心，还是最大连通域中心；
- 是否允许旋转校正；
- 是否保留字形原始纵横比。

如果这一部分定义不清，后续骨架差异和 EDT 差异都会被平移/缩放误差污染。

#### 2.2.2 预处理规则不完整

目前只提到“二值化 + 去噪”，但没有约束：

- 输入是白底黑字还是黑底白字；
- 是否存在拍照阴影、压缩噪声、背景纹理；
- 去噪采用中值滤波、形态学开运算还是连通域筛除；
- 是否需要孔洞填充。

对于图像原型，这些规则必须标准化，否则同一算法在不同图片上结果会波动较大。

#### 2.2.3 骨架指标需要更具体

`Hausdorff / Chamfer` 都可行，但要提前说明主指标和辅助指标。

建议：

- 主指标：对称 Chamfer Distance（更稳健，不容易被单个离群点主导）；
- 辅助指标：Hausdorff Distance（用于显示最大偏移风险）；
- 可视化层：局部最近邻距离图。

原因是 Hausdorff 对孤立异常像素过于敏感，单独使用可能使原型不稳定。

#### 2.2.4 EDT 差异容易受边界和配准误差影响

`|EDT_user - EDT_expert|` 的定义直观，但前提是两张图已在同一空间坐标系中良好对齐。

需要补充：

- EDT 是对前景计算还是对背景计算；
- 热力图是否只在前景联合区域内显示；
- 是否需要做归一化（例如除以最大笔画半径）；
- 如何避免空白区域大面积主导颜色范围。

否则热力图可能看起来“很红”，但本质上只是配准误差，而不是字形差异。

#### 2.2.5 可视化需求还可再收敛

当前 Visualization 模块包含：图层切换、透明度、缩放、hover 等功能，整体合理，但首版原型不宜做得过重。

建议按 MVP 原则分层：

- 第一阶段：静态结果 + 网页展示；
- 第二阶段：图层切换 + 透明度控制；
- 第三阶段：hover 数值、局部指标、结果导出。

这样可以避免一开始过度投入前端交互，影响核心算法验证。

## 3. 建议补充的项目定义

在正式实施前，建议将以下内容写入后续设计文档或 README：

### 3.1 输入约束

- 输入格式：`png/jpg`
- 输入颜色：自动转灰度
- 默认前景：深色字形、浅色背景
- 单图仅包含一个主字形
- 允许的分辨率范围与最小尺寸

### 3.2 标准画布与坐标规范

- 标准尺寸：如 `256x256`
- 保持纵横比缩放
- 前景居中贴到统一画布
- 坐标原点、数组方向、显示方向保持一致

### 3.3 差异指标规范

至少输出以下指标：

- `symmetric_chamfer`
- `hausdorff`
- `mean_edt_diff`
- `max_edt_diff`
- `overlap_ratio`（可选）

### 3.4 原型验收标准

建议定义最低可验收目标：

1. 能稳定读取两张图片；
2. 能输出标准化后的二值图；
3. 能输出骨架叠加图；
4. 能输出 EDT 热力图；
5. 能在本地网页中切换查看结果；
6. 对示例图片运行不报错。

## 4. 项目执行计划（阶段划分）

以下计划按“先验证算法闭环，再增强交互与工程化”的原则制定。

---

## Phase 0：项目定义与样例基线

### 目标

明确输入约束、目录结构、技术选型和最小验收标准，为后续开发建立统一基线。

### 需要构建的内容

- 项目目录结构
- `requirements.txt` 或 `pyproject.toml`
- 样例数据目录（如 `data/examples/`）
- 配置文件（如 `config.yaml`）
- README 中的运行说明与验收标准

### 任务清单

1. 明确输入输出文件规范；
2. 确定标准画布尺寸与坐标约定；
3. 确定依赖库版本；
4. 放入一组 Expert/User 示例图片；
5. 确定 MVP 功能边界。

### 阶段产出

- 可运行的项目骨架
- 明确的技术与数据规范

---

## Phase 1：预处理与空间归一化

### 目标

把任意输入图像转换成可用于结构比较的统一表示，这是整个系统最关键的基础阶段。

### 需要构建的内容

- 图像读取模块
- 灰度化与二值化模块
- 去噪模块
- 前景提取与裁剪模块
- 统一缩放与居中对齐模块
- 调试可视化输出

### 任务清单

1. 读取图像并统一颜色空间；
2. 实现阈值分割（固定阈值或 Otsu）；
3. 实现小连通域去除与形态学清理；
4. 提取外接框并裁剪；
5. 按保持纵横比规则缩放至标准画布；
6. 将前景贴到统一大小画布中央；
7. 输出中间结果图用于核查。

### 阶段产出

- `preprocess(image) -> normalized_binary`
- 标准化前后对比图

### 风险控制

- 优先避免“看起来对齐了，实际上坐标不一致”；
- 保证 Expert/User 使用完全相同的预处理逻辑。

---

## Phase 2：骨架提取与结构差异分析

### 目标

完成结构层的核心表达与量化，生成骨架叠加结果和结构差异指标。

### 需要构建的内容

- 骨架提取函数
- 骨架点集表示
- Chamfer/Hausdorff 计算函数
- 骨架叠加图生成器
- 结构指标输出模块

### 任务清单

1. 对归一化二值图执行 `skeletonize`；
2. 将骨架转换为点坐标集合；
3. 计算对称 Chamfer 距离；
4. 计算 Hausdorff 距离；
5. 生成 Expert/User 双色叠加图；
6. 输出结构指标 JSON 或字典。

### 阶段产出

- `extract_skeleton(binary)`
- `compare_skeleton(expert_skel, user_skel)`
- `render_skeleton_overlay(...)`

### 验收标准

- 能稳定生成双骨架图；
- 指标数值与视觉差异基本一致。

---

## Phase 3：EDT 墨迹场与热力差异分析

### 目标

建立笔画厚度/边界层面的差异表示，补足单纯骨架比较无法反映的粗细问题。

### 需要构建的内容

- EDT 计算模块
- EDT 差异场模块
- 差异归一化模块
- 热力图渲染模块
- 区域统计模块

### 任务清单

1. 对归一化二值图计算 EDT；
2. 计算 `abs(edt_user - edt_expert)`；
3. 限制热力图显示区域（建议前景联合区域）；
4. 设计颜色映射和显示范围；
5. 输出均值、最大值等统计量；
6. 生成热力图图像。

### 阶段产出

- `compute_edt(binary)`
- `compute_edt_diff(edt_expert, edt_user)`
- `render_heatmap(diff_map)`

### 风险控制

- 防止背景区域掩盖真实差异；
- 防止未对齐造成虚假高差异。

---

## Phase 4：结果聚合与本地脚本化运行

### 目标

把预处理、骨架比较、EDT 比较串成一条可重复执行的 pipeline，先完成命令行闭环。

### 需要构建的内容

- 主流程脚本
- 参数接口（输入路径、输出路径）
- 结果保存目录
- 汇总报告（图片 + 指标）

### 任务清单

1. 编写 `main.py` 或 `run_pipeline.py`；
2. 支持输入两张图像路径；
3. 自动输出：
   - 标准化图
   - 骨架叠加图
   - 热力图
   - 指标结果文件
4. 增加日志输出；
5. 用示例图片跑通全流程。

### 阶段产出

- 一键运行命令
- 完整输出目录结构

### 验收标准

- 对示例输入可稳定执行到最终结果；
- 输出内容命名清晰、可复现。

## 5. 推荐的目录结构

建议后续项目按以下结构组织：

```text
Character_Glyph_Contrast/
├── project.md
├── plan.md
├── README.md
├── requirements.txt
├── data/
│   └── examples/
├── outputs/
├── src/

│   ├── skeleton.py
```

---

## 6. 建议的开发优先级

推荐按以下顺序推进，而不是同时开发所有功能：

1. **先做预处理与对齐**；
2. **再做骨架叠加结果**；
3. **再做 EDT 热力图**；
4. **最后接入 Dash 页面**。

原因：

- 如果对齐不稳，后续所有比较都会失真；
- 如果核心结果图还不可靠，先做网页意义不大；
- 原型阶段优先验证“结果是否可信”，再优化“展示是否炫”。

---

## 7. 最终建议

综合判断，本项目**适合立即启动原型开发**，但应将其定义为：

> 一个面向静态字形图像的、以几何匹配为核心的差异可视化原型系统。

最重要的实施原则是：

- 先统一输入与坐标；
- 先验证结构层结果；
- 再补充墨迹层差异；
- 最后构建交互式网页。

如果严格按本计划推进，项目可以在较短周期内形成一个可演示、可解释、可扩展的首版系统。
